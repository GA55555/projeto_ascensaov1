# Relatório de Engenharia — Migração de Fichas (PDF → JSONB) & Hardening de Auth

**Data:** 2026-06-17
**Branch:** `sandbox`
**Âmbito:** 6 missões sequenciais (Refatoração Auth → Fase 1–4 das Fichas Nativas → Bug Bash → Saneamento de Sessão)
**Estado de sintaxe:** ✅ Todos os ficheiros tocados passam `node -c`.
**Estado funcional real (DB/browser):** ⚠️ **NÃO verificado end-to-end** — nenhum teste correu contra Postgres real nem browser. Ver secção 7.

## 1. Sumário Executivo

O projeto migrou o módulo "Gaveta de Fichas" de **upload de PDFs físicos** para **fichas nativas em coluna JSONB** (`gaveta_fichas.dados_ficha`), e em paralelo endureceu a camada de autenticação (MVC + Zod + Cookies HttpOnly + verificação de sessão server-side).

As 6 missões foram executadas e validadas **ao nível de sintaxe e de lógica isolada** (schemas Zod, render headless dos templates, reconstrução do coletor). O que **não** foi possível validar: o comportamento real contra o PostgreSQL e o browser. Como há bugs em aberto, a **Secção 7 (Dependências & Riscos)** é a parte mais importante deste documento.

## 3. Fluxo Atual do Sistema (como deveria funcionar)

### 3.1 Autenticação
1. `POST /auth/login` (validado por `loginSchema`) → `AuthController.login` → grava cookie **HttpOnly** `m20_token` + devolve `{ usuario }` no JSON.
2. Frontend (`login.html`) grava `m20_user`/`m20_tema` no `localStorage` (apenas para UX/guards) e redireciona.
3. Rotas protegidas: middleware `verificarToken` lê **`req.cookies.m20_token`** (fallback para header `Authorization` legado) e injeta `req.usuario`.
4. `GET /auth/verificar` → 200 se cookie válido, 401 caso contrário. Usado por `verificarSessao()`.
5. `POST /auth/logout` → `res.clearCookie('m20_token')`.

### 3.2 Fichas (CRUD JSONB)
- `GET /gaveta/fichas` → `listarFichas` (`SELECT id, nome, tipo, dados_ficha, criado_em`).
- `POST /gaveta/fichas` (`criarFichaSchema`) → `criarFicha` → `INSERT ... dados_ficha` (sistema + versão dobrados no JSONB).
- `PUT /gaveta/fichas/:fichaId` (`atualizarFichaSchema`) → `atualizarFicha` → `UPDATE ... SET dados_ficha=$1, nome=COALESCE($2,nome)`.
- `DELETE /gaveta/fichas/:fichaId` → `deletarFicha` → só `DELETE` (sem ficheiros físicos).

### 3.3 Editor de Fichas (Frontend)
`carregarFichas()` → cache `fichasCache` + render da lista (badge de versão + botão "Abrir/Editar") → `abrirEditorFicha(id)` escolhe o template por `dados_ficha.sistema` (`renderizarFichaDnD` / `renderizarFichaMago`) → modal fullscreen → `coletarDadosFicha()` varre `[data-path]` reconstruindo o objeto aninhado → `salvarEdicaoFicha` (PUT) ou `salvarComoNovaVersao` (POST).

## 7. ⚠️ Dependências Técnicas & Riscos de Bug (LER PRIMEIRO)

> Esta secção lista o que pode estar a causar os teus bugs. Ordenado por severidade.

### 7.1 🔴 [EXTERNO/CRÍTICO] Esquema real do PostgreSQL não verificado
Todo o módulo assume que o DBA executou a migração para:
```
gaveta_fichas ( id, usuario_id, nome, tipo, dados_ficha JSONB, criado_em )   -- SEM url_arquivo
```
Se a coluna `dados_ficha` **não existir** ou `url_arquivo` **ainda existir NOT NULL**, então `criarFicha`/`atualizarFicha`/`listarFichas` devolvem **500** (`column ... does not exist` ou `null value violates not-null`).
**Como confirmar:** `\d gaveta_fichas` no psql. Este é o suspeito nº1.

### 7.2 🔴 [CRÍTICO] Tipo do `id` vs validação `z.string()`
No Bug Bash troquei `fichaId` de `.uuid()` para `z.string()` assumindo **IDs sequenciais numéricos**. Se o `id` for `SERIAL`/`INT`, o frontend envia-o como string na URL e o `WHERE id = $3` faz cast implícito — OK. Mas se houver mismatch de tipo (ex.: `id` é `uuid` e chega algo não-uuid), o Postgres dá **erro de cast 22P02** (`invalid input syntax`). **Confirmar o tipo real da PK.**

### 7.3 🟠 [ALTO] `dados_ficha` como objeto JS no parâmetro `pg`
`criarFicha`/`atualizarFicha` passam um **objeto JS** (`dados`) como parâmetro para uma coluna JSONB. O `node-postgres` serializa objetos via `JSON.stringify` automaticamente — **deve** funcionar. Mas se a coluna estiver tipada como `json`/`text` em vez de `jsonb`, ou se houver um driver/tipagem custom, pode gravar `"[object Object]"`. **Confirmar que a coluna é `jsonb`** e que uma linha gravada contém JSON real.

### 7.4 🟠 [ALTO — SEGURANÇA] `deletarFicha` não filtra por `usuario_id`
```js
await pool.query('DELETE FROM gaveta_fichas WHERE id = $1', [fichaId]);
```
Falta `AND usuario_id = $2`. Um utilizador autenticado pode apagar a ficha de **outro** utilizador conhecendo o `id` (IDOR — viola Regra 3.3). Não causa o crash atual, mas **deve ser corrigido**:
```js
await pool.query('DELETE FROM gaveta_fichas WHERE id = $1 AND usuario_id = $2', [fichaId, req.usuario.id]);
```

### 7.5 🟡 [MÉDIO] Conflito de definições de `escapeHTML` (ordem de carregamento)
Existem **3** definições: `api.js` (com guard null→`''`), `utils.js` (criado agora, **sem** guard → `escapeHTML(null)` devolve a string `"null"`), e uma local em `escudo_narrador.js`. No `index.html` a ordem é `api.js` → `utils.js`, então **a versão menos defensiva vence**. Hoje não quebra porque os renders usam `|| ''`, mas é uma regressão subtil. **Recomendação:** alinhar `utils.js` ao guard de null do `api.js`.

### 7.6 🟡 [MÉDIO] Toast de erro espúrio em página deslogada
`API.fetch` em 401/403 **já** redireciona e faz `throw`; o `catch` dele chama `mostrarToast("Erro de comunicação...")`. Como `verificarSessao` usa `API.fetch`, um utilizador sem sessão vê um **toast de erro + erro no console** antes do redirect. Não é crash, mas parece bug. Considerar tratar 401 silenciosamente em `verificarSessao`.

### 7.7 🟡 [MÉDIO] Cruft de dados entre modal simples e templates ricos
O modal "Criar Nova Ficha" grava `dados_ficha = { sistema, hp_atual, hp_maximo }`. Para **Mago**, o template não tem campos de HP (tem Vitalidade), então ao editar/salvar a ficha fica com `hp_atual/hp_maximo` órfãos + a estrutura M20 completa. Não quebra, mas suja o JSONB. Para D&D é consistente (o template tem HP).

### 7.8 🟢 [BAIXO] `verificarSessao` só protege 2 páginas
Apenas `escudo_narrador.js` e `config_perfil.html` chamam `verificarSessao()`. As restantes ainda dependem do guard `localStorage.getItem('m20_user')` + 401 do backend. A proteção real é server-side (OK), mas é inconsistente com o pedido da Missão 6.

### 7.9 🟢 [BAIXO] `validate.js` usa `err.errors` (depreciado no Zod v4)
Funciona (getter de compatibilidade), mas pode partir num major futuro do Zod. Migrar para `err.issues` quando possível.

## 8. Checklist de Depuração Sugerida

1. **DB primeiro** (7.1/7.2/7.3): `\d gaveta_fichas` — confirmar `dados_ficha jsonb` existe, `url_arquivo` não existe, e o tipo da PK `id`. Inserir uma ficha pela UI e fazer `SELECT dados_ficha FROM gaveta_fichas` para ver se gravou JSON real ou `"[object Object]"`.
2. **Network tab**: abrir DevTools → Network → reproduzir o bug. Ver o **status real** (401? 500? 400?) e o corpo do erro. O backend devolve `{ erro: ... }` com mensagem clara.
3. **Cookie**: DevTools → Application → Cookies → confirmar que `m20_token` existe e é `HttpOnly` após login. Se `secure:true` e estiver em `http://` (não-localhost), o browser **não envia o cookie** → 401 em tudo. (`secure` só liga em `NODE_ENV==='production'`.)
4. **Console**: confirmar ausência de `ReferenceError`. Se aparecer `mostrarToast is not defined` ou `API is not defined`, é **ordem de `<script>`** nalgum HTML.
5. **Corrigir 7.4** (IDOR no delete) independentemente do bug atual.
