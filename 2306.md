## 7.0 Decisões de Game Design travadas (Narrador, 23/06)
1. **Origem das imagens = UPLOAD (reusar pipeline existente).** Sem URLs externas na v1 → elimina superfície de XSS/CSS-injection, links quebrados e privacidade.
2. **Forma do avatar = preenche o card arredondado atual.** A forma REDONDA (token VTT) vira variação de **Tema** (Fatia 3 Mapa Tático), separando "imagem" de "geometria".
3. **Ordem = Fatia 1 (Fundo) → Fatia 2 (Avatares) → Fatia 3 (Temas).**

## 7.1 Premissas verificadas (o que existe de verdade)
- **Camadas do board:** `#board-canvas` (estático, segura grid/dots em background-position parallax + handlers de pan/zoom) › `#board-world` (sofre `transform: translate+scale`). **Conclusão:** uma imagem que se move/zooma com os cards tem de viver **dentro de `#board-world`** (não no canvas), na camada mais baixa (abaixo das zonas, que são z-index 0).
- **Pipeline de upload já existe e é 100% conforme** (Regras 6.3/6.5/6.6): `config/upload.js` (multer `memoryStorage`, whitelist mimetype jpeg/png/webp), `controllers/midiaController.js` (Sharp → `.webp({quality:80})`, resize 1920×1080 inside, nome `crypto.randomBytes(16).hex`), rota `POST /upload/:tipo` (`routes/midiaRoutes.js`) com whitelist `pastasPermitidas=['capas','posts','avatares','cards','social']`, devolve `{ urls:['/uploads/tipo/hash.webp'] }`. Há `avatares` (serve à F2); **não há `fundos`** → F1 precisa registrar `'fundos'` no array + criar `public/uploads/fundos/` (código + filesystem, **não é DDL**).
- **Avatar de Entidade = SEM DDL:** `world_nodes` tem `dados` (jsonb) → `dados.avatar_url` cabe lá (Regra 4.1 OK).
- **Avatar de Núcleo = BLOQUEIO de DDL:** `entidade_nucleos` só tem `nome,tipo,criado_em,id,cronica_id` — **sem jsonb, sem coluna de imagem**. Brasão global de facção exigiria **DDL (DBA, Regra 4.1)**.
- **Tema = Paradigma 5 já em uso:** `document.documentElement.classList.add(tema)` (`api.js`) + `.tema-pixel-16bit` (scope override em `global_ui.css`). F3 encaixa como classe **escopada ao board** (`#board-canvas`), sem tocar o tema global.

## 7.2 Riscos / armadilhas a blindar (transversais)
- **R1 — Zod descarta chaves desconhecidas:** todo campo novo (`fundoImagem`, `tema`, `celulas[].avatar_url`, `dados.avatar_url`) **DEVE** ser declarado no schema correspondente, senão o save o remove silenciosamente (mesma lição da flag `travada`).
- **R2 — `z.string().url()` rejeita `/uploads/...`** (caminho relativo). Como só usamos uploads locais, validar com **regex/refine** que exige prefixo `^/uploads/` (não `.url()`). Bloqueia também `javascript:`/`data:`/externos por construção.
- **R3 — XSS/CSS-injection:** render via `<img src="${escapeHTML(url)}">` (Regra 6.1), **nunca** `background-image:url(${cru})` (quebra de CSS). `onerror` → fallback gracioso (ícone/placeholder).
- **R4 — Ficheiros órfãos (Regra 6.6):** trocar/remover um fundo ou avatar deixa o `/uploads/...` antigo órfão. O save do board só guarda a string → **não há hook de `fs.unlink`**. v1: ou (a) aceitar órfãos e **declarar a dívida**, ou (b) endpoint de limpeza comparando url antiga↔nova. Recomendado v1: declarar a dívida; tratar numa fatia de higiene.
- **R5 — Regra 2.7 (save manual):** fundo/avatar/tema entram no `boardState`/`dados` e só persistem no **Salvar** explícito; nada de auto-save.
- **R6 — Coerência visual (Regra 2.5):** geometria dos temas (polaroid/token/cortiça) só por **classe CSS escopada**; zero inline estético; cores por token; `lucide.createIcons()` após qualquer injeção.

## 7.3 Fatia 1 — Fundo Personalizado (Mapa/Stencil) · *PRÓXIMA*
**Objetivo:** o Narrador faz upload de um mapa/textura e ele vira a camada de fundo do tabuleiro, movendo/zoomando junto com os cards.
**Modelo:** `boardState.fundoImagem = { url, x, y, w, h } | null` (rect em coords de mundo). Zod: objeto opcional com `url` validada por refine `^/uploads/` (R2) + `x,y,w,h` finitos (R1).
**Render:** em `renderBoard`, emitir `<img class="board-imagem-fundo" src="${escapeHTML(url)}" style="left/top/width/height">` como **primeiro** filho do corpo; CSS `z-index:-1` dentro do `#board-world` → abaixo das zonas (z0) e de tudo, **acima** da grade do canvas (filho pinta sobre o fundo do pai). `pointer-events:none` por padrão (não rouba clique/pan).
**Upload (reusar):** registrar `'fundos'` em `pastasPermitidas` + criar `public/uploads/fundos/`; reusar o helper de upload do front (padrão de `profile.js`/`escudo_narrador.js`), idealmente isolado num `MidiaApi`/`mundoApi` (Regra 2.4). Controle na `.board-toolbar` ("Imagem de Fundo": enviar / remover).
**Sub-fatias:** (1a) modelo+Zod+upload+render fixo; (1b) posicionar = arrastar+redimensionar (reusar o padrão de `arrastarPorPonteiro`/shape, zoom-aware, `pointerup`+`pointercancel`); (1c) niceties: opacidade, auto-esconder grade quando há imagem, "remover" (+ nota R4 órfãos).
**Riscos locais:** R2, R3, R4, R1. **Sem DDL, sem backend novo** além de 1 linha no whitelist + a pasta.

## 7.5 Fatia 3 — Motor de Temas (por último; compõe F1+F2)
**Modelo:** `boardState.tema ∈ {'esquema'(default),'investigacao','mapa-tatico'}` (Zod enum, R1). **Aplicação:** classe escopada no `#board-canvas` (`board-tema-*`), Paradigma 5 — **não** no `documentElement` (não contaminar o app). Pura apresentação (não muta domínio).
**Temas:** *Esquema* = atual (sem overrides). *Investigação* = canvas cortiça (textura/token de cor), cards "polaroid" (borda branca inferior + fonte manuscrita no nome), linhas "lã vermelha" (stroke/efeito). *Mapa Tático* = usa `fundoImagem` (F1) + cards viram **tokens redondos** (precisa do avatar F2; texto oculto). **Dependências:** F1 (fundo) e F2 (avatar) → por isso é a última.
**Risco-chave:** o HTML do card deve ser **theme-agnostic** — só CSS escopado faz a metamorfose (polaroid/token). Garantir um "slot" de avatar (vindo da F2) que o CSS do tema reformata. Zero inline (R6).
